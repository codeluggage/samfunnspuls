# Refresh RK Context

Use this skill when the upstream DesignSystem guidance has changed and the template needs a controlled refresh.

## Inputs

- Upstream `AI_DESIGN_SYSTEM_GUIDE.md`
- Upstream `metadata.json`
- Upstream `ai-context.manifest.json`
- Installed `rk-design-tokens/design-tokens-build/theme.css` (mirrored into the `rk-brand-design` skill so static HTML mocks have every named token).

## Default Flow

1. Refresh the checked-in upstream context artifacts.
2. Mirror the installed `rk-design-tokens` theme into `.agents/skills/rk-brand-design/tokens/theme.css` so the brand skill stays in sync with the production token contract.
3. Generate a drift report with:
   - safe refreshes
   - rule or policy conflicts
   - follow-up work candidates
   - issue-ready recommendations
4. Sync shared skills and rules into tool-specific outputs.
5. Stop if the refresh implies standards decisions instead of a straightforward context update.

## Commands

```bash
npm run guide:refresh
npm run guide:refresh -- --source=local --designsystem-root=../DesignSystem
```

## Guardrails

- Do not auto-clean application code as part of the refresh.
- Treat drift as reviewable information first.
- Preserve target-side tool customizations unless the engineer explicitly forces an overwrite.
- The `rk-brand-design` token mirror is regenerated, not hand-edited. The hand-curated `mocks/colors_and_type.css` subset is **not** regenerated by refresh — drift between it and the upstream theme is a review point, not a safe refresh.

## Cross-references

- `rk-brand-design` — receives the regenerated `tokens/theme.css` mirror.
- `designsystem-proof` — proof reports reference the brand non-negotiables; if a refresh changes those, proof reports need re-running.
